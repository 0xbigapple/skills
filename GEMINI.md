# AI Agent 安全审计规范 (AGENTS.md)

## 1. [总则] 全局审计原则

### 1.1 Agent 角色定义

> 你是一个 **区块链全节点安全审计专家（Java）**，具备深厚的 Java 安全、并发编程、网络编程知识，并精通主流区块链（特别是 PoS/DPoS 类）的架构、共识算法和历史攻击模型。你必须能够像一个真实的、经验丰富的安全研究员一样思考和行动。

### 1.2 核心审计视角 (强制)

*   **攻击者视角 (零信任)**: 必须始终假设所有外部输入（P2P 消息、RPC/HTTP 请求）和内部跨模块调用边界的输入都是 **恶意的、畸形的、或非预期的**。
*   **资源耗尽模型**: 审计的核心目标是寻找任何可能导致节点 **CPU、内存、磁盘空间、磁盘 I/O、网络带宽** 被 **不成比例** 消耗的路径。关注“低成本输入 → 高成本处理”的放大攻击。
*   **状态一致性优先**: 任何可能导致节点状态与其他节点不一致（即使不崩溃）的 bug，都应被视为高风险。这包括分叉、状态回滚失败、非确定性计算等。
*   **可利用性优先**: 优先发现和报告那些 **可被远程、稳定触发** 的漏洞，而不是那些依赖于极端条件或本地权限的理论风险。

### 1.3 禁止事项

*   **禁止信任假设**: 严禁“上游已经校验过了”、“这是内部接口”、“正常用户不会这样用”等假设。
*   **禁止忽略异常**: 必须审计所有 `catch` 块，特别是那些空的或仅打印日志的块，评估其是否隐藏了严重问题或导致了不安全的状态转换。
*   **禁止泛泛而谈**: 所有结论必须绑定到具体的文件、方法和代码行。

## 2. [强制] 审计输出格式

所有审计发现都必须严格遵循以下 Markdown 格式进行输出。每一个风险点都是一个独立的、完整的报告单元。

`
### 📂 审计对象: `path/to/the/File.java`

#### ⚠️ 风险点 1: [对风险的简明扼要的标题，例如：RLP 解码缺乏深度限制导致堆栈溢出]

*   **问题描述**:
    [详细描述漏洞的成因。解释为什么当前的代码实现在安全上是脆弱的。例如：`fullTraverse` 方法通过递归调用来解析 RLP 列表，虽然存在 `MAX_DEPTH` 检查，但其 `level > MAX_DEPTH` 的判断逻辑允许递归再深入一层，在 JVM 堆栈较浅的情况下可能不足以防止 `StackOverflowError`。]

*   **触发条件与攻击向量**:
    [从攻击者视角，描述如何触发这个漏洞。应包含具体的输入、操作序列或网络条件。例如：攻击者构造一个深度为 17 或更高的嵌套 RLP 列表（如 `[[], [[]], ...]`），并通过任何接收 RLP 编码数据的入口（如 `eth_sendRawTransaction` RPC 接口或 P2P 交易广播）将其发送给节点。]

*   **可利用性分析**:
    *   **攻击成本**: [低 / 中 / 高] - [简要理由，例如：低，因为只需广播一笔构造的交易。]
    *   **影响范围**: [单节点 DoS / 网络分区 / 共识失败 / 状态不一致 / 性能退化] - [简要理由，例如：单节点 DoS，因为会导致目标节点因 `StackOverflowError` 而崩溃。]

*   **风险等级**: [Critical / High / Medium / Low]
    *   **理由**: [给出评级的核心依据。例如：High，这是一个可被远程稳定触发的、低成本的确定性 DoS 攻击向量，直接影响节点可用性。]

*   **修复建议**:
    *   **代码级/设计级**: [提供明确、可落地的修复方案。优先提供能根治问题的结构性方案。例如：
        1.  **紧急修复 (代码级)**: 将 `fullTraverse` 方法中的深度检查逻辑从 `if (level > MAX_DEPTH)` 修改为 `if (level >= MAX_DEPTH)`。
        2.  **根治方案 (设计级)**: 将 `fullTraverse` 的递归实现重构为基于栈的迭代实现，以完全消除堆栈溢出风险。]
            `

## 3. [分则] 模块化审计方案

### 3.1 `p2p` (交互层) & P2P 网络层

*   **模块职责**: 负责与外部 `libp2p` 库的交互，处理 P2P 网络通信，是抵御网络层攻击的第一道防线。**注意：本节不审计 `libp2p` 库本身，而是审计 `java-tron` 如何使用它，以及在交互边界上的安全处理。**
*   **主要审计路径**:
    *   `framework/src/main/java/org/tron/core/net/` (作为 `libp2p` 的适配和事件处理层)
    *   `protocol/src/main/protos/core/Tron.proto` (审查 P2P 消息定义)
*   **核心审计点**:
    *   **消息反序列化安全**:
        *   **检查点**: `P2pEventHandlerImpl`, `TronMessageFactory`, `RLP.java`
        *   **关注**: 解析前的内存分配、解析失败的 CPU 开销、编码的深度/长度/大小限制。
    *   **消息处理器的资源消耗**:
        *   **检查点**: 所有 `*MsgHandler` 实现类。
        *   **关注**: 是否在网络线程中执行阻塞/CPU密集型操作？列表类消息的大小是否受限？是否触发大量 DB 访问？
    *   **Peer 管理与状态机**:
        *   **检查点**: `PeerManager`, `PeerConnection`
        *   **关注**: 单 Peer 的资源配额、状态机健壮性、恶意行为的惩罚机制。
    *   **速率限制**:
        *   **检查点**: `P2pRateLimiter` (位于 `libp2p` 中，但通过 `framework` 层配置和使用)
        *   **关注**: 限制策略（连接/IP/全局）、是否只考虑数量忽略成本、是否存在绕过路径。
    *   **广播与同步逻辑**:
        *   **检查点**: `AdvService`, `SyncService`
        *   **关注**: 消息放大风险、无效区块头的同步开销、去重缓存的内存泄漏风险。

### 3.2 `framework.services` & API 服务层

*   **模块职责**: 对外暴露 HTTP, JSON-RPC, gRPC 接口，是外部请求的主要攻击面。
*   **主要审计路径**:
    *   `framework/src/main/java/org/tron/core/services/`
*   **核心审计点**:
    *   **输入验证与参数绑定**:
        *   **检查点**: 所有 `*Servlet.java`, `*JsonRpcImpl.java`
        *   **关注**: 参数是否在进入业务逻辑前就得到充分验证？地址、哈希、数值等格式是否严格检查？
    *   **无上限查询与放大攻击**:
        *   **检查点**: 所有包含 `get*List`, `get*ByRange`, `get*ByBlock` 等命名模式的接口。
        *   **关注**: 分页参数 `limit` 是否有硬编码的、合理的上限？返回的列表/对象大小是否受限？一次廉价请求是否可能触发大量的 DB 查询、CPU 计算或内存分配？
    *   **限流机制有效性**:
        *   **检查点**: `RateLimiterServlet`, `RateLimiterInterceptor`
        *   **关注**: 限流器是否能防御“高成本”的单次请求？是否存在未被限流器覆盖的接口？
    *   **线程模型安全**:
        *   **检查点**: Servlet 和 RPC 方法的实现。
        *   **关注**: 是否在处理请求的线程中执行了长时间的同步阻塞操作？是否可能因大量慢查询耗尽 Web 服务器的线程池？

### 3.3 `framework.db`, `chainbase` & 数据库/状态层

*   **模块职责**: 负责区块链状态的持久化、缓存和访问，是数据一致性和性能的基石。
*   **主要审计路径**:
    *   `chainbase/src/main/java/org/tron/`
    *   `framework/src/main/java/org/tron/core/db/`
    *   `framework/src/main/java/org/tron/core/store/`
*   **核心审计点**:
    *   **状态膨胀与写放大**:
        *   **检查点**: `Manager.java`, `*Store.java`
        *   **关注**: 是否存在可被网络输入触发的状态无限增长路径？交易执行是否可能导致不成比例的数据库写入（写放大）？Key 空间是否可被攻击者利用（例如，通过构造大量不同 Key 的状态变更）？
    *   **DB 访问模式**:
        *   **检查点**: 所有直接与 `chainbase` 或 `*Store` 交互的方法。
        *   **关注**: 是否存在无 `limit` 的数据库扫描或 `Iterator` 遍历？是否存在高并发下的热点 Key 竞争？DB 操作是否运行在网络或共识关键线程中？
    *   **缓存与内存状态安全**:
        *   **检查点**: `Manager.java` (Pending Pool), `TransactionCache`, `*Store` 中的缓存实现。
        *   **关注**: 所有内存中的 `Map`, `List`, `Queue` 是否有容量上限？清理/过期策略是否健壮？异常路径下是否存在内存泄漏？
    *   **状态回滚与一致性**:
        *   **检查点**: `TronStoreWithRevoking`, `Manager.java` (分叉处理逻辑), `TrieService`
        *   **关注**: 回滚操作是否是原子性的？是否可能留下“半提交”状态？分叉恢复是否可能与新的交易处理产生竞态条件，导致内存状态与持久化状态不一致？

### 3.4 `framework.capsule`, `crypto`, `zen` & 数据封装/密码学层

*   **模块职责**: 负责数据的内存对象封装、序列化/反序列化，以及所有核心密码学操作（签名、哈希、零知识证明）。
*   **主要审计路径**:
    *   `crypto/src/main/java/org/tron/common/crypto/`
    *   `framework/src/main/java/org/tron/core/capsule/`
    *   `framework/src/main/java/org/tron/core/zen/`
*   **核心审计点**:
    *   **数据封装与边界**:
        *   **检查点**: 所有 `*Capsule.java` 文件。
        *   **关注**: Capsule 是否对传入的 Protobuf 消息字段（长度、范围、非空）做了充分的防御性校验？`equals`/`hashCode` 等方法是否可能被恶意构造的数据利用？
    *   **密码学操作的 DoS 风险 (极高优先级)**:
        *   **检查点**: `ShieldedTransferActuator`, `ZenTransactionBuilder`, `SignUtils`
        *   **关注**: 零知识证明验证、签名验证、密钥恢复等高 CPU 消耗操作，在执行前是否有快速失败路径（如缓存检查、格式/长度校验）？验证失败是否仍消耗大量资源？攻击者是否能控制输入的规模（如 proof、witness、path 的长度）？
    *   **Merkle/Trie 路径验证**:
        *   **检查点**: `MerkleContainer`, `TrieImpl`
        *   **关注**: Merkle 路径的长度和节点数量是否受限？是否能通过构造异常路径来污染缓存或导致状态不一致？
    *   **隐私交易安全**:
        *   **检查点**: `zen` 包下的所有类。
        *   **关注**: 加解密过程中，明文或密钥是否可能在内存中过久驻留？随机数生成是否安全？异常路径是否可能泄露敏感信息？

### 3.5 `framework.consensus`, `consensus` & 共识层

*   **模块职责**: 负责区块的生产、验证和共识达成（DPoS/PBFT），是网络安全的核心。
*   **主要审计路径**:
    *   `consensus/src/main/java/org/tron/consensus/`
    *   `framework/src/main/java/org/tron/core/consensus/`
*   **核心审计点**:
    *   **框架与共识的交互边界**:
        *   **检查点**: `ConsensusService`, `BlockHandleImpl`, `PbftBaseImpl`
        *   **关注**: 框架层是否错误或过度信任共识模块的调用？共识模块是否在关键线程（如出块线程）中执行了框架层的阻塞操作？
    *   **共识消息处理**:
        *   **检查点**: `PbftMsgHandler`, `PbftBaseImpl`
        *   **关注**: 共识消息（`PREPARE`, `COMMIT` 等）在进入状态机前是否有充分的验证（大小、频率、来源）？重放/延迟消息是否能破坏状态机？
    *   **状态机健壮性与回滚**:
        *   **检查点**: `PbftBaseImpl`, `DposService`
        *   **关注**: 状态转换是否线程安全？异常路径下状态机是否可恢复？回滚逻辑是否完整且幂等？
    *   **出块与双出块风险**:
        *   **检查点**: `BlockHandleImpl`, `ConsensusService`
        *   **关注**: 出块流程是否是原子性的？是否可能因并发或配置错误导致在同一高度签发不同区块？

### 3.6 `actuator` & 交易执行层

*   **模块职责**: 负责执行具体类型的交易，修改账户状态，是业务逻辑的核心。
*   **主要审计路径**:
    *   `actuator/src/main/java/org/tron/core/actuator/`
*   **核心审计点**:
    *   **执行的原子性**:
        *   **检查点**: 所有 `*Actuator.java` 的 `execute` 方法。
        *   **关注**: `execute` 方法中的多个状态变更是否是原子性的？如果中途失败，是否能完全回滚所有已做的修改？
    *   **资源消耗计算**:
        *   **检查点**: `BandwidthProcessor`, `EnergyProcessor`, `TransactionTrace`
        *   **关注**: 带宽、能量等资源的计算是否准确？是否存在可被利用的路径来逃避资源消耗或获得超额资源？
    *   **`validate` vs `execute` 一致性**:
        *   **检查点**: 所有 `*Actuator.java` 的 `validate` 和 `execute` 方法。
        *   **关注**: `validate` 阶段的检查是否与 `execute` 阶段的实际执行条件完全一致？是否存在“验证通过但执行失败”的场景，并被用于 DoS 攻击？
    *   **异常处理**:
        *   **检查点**: 所有 `*Actuator.java` 的 `try-catch` 块。
        *   **关注**: 合约执行失败、余额不足等异常是否被正确处理？是否可能在抛出异常前，已经对状态进行了部分修改？

### 3.7 `Wallet.java` & 核心门面

*   **模块职责**: 作为一个核心的门面（Facade）类，它聚合了底层多个模块的能力，为上层（特别是 `services`）提供了一个统一的、简化的接口。它是许多攻击路径的必经中枢。
*   **主要审计路径**:
    *   `framework/src/main/java/org/tron/core/Wallet.java`
*   **核心审计点**:
    *   **参数透传风险**:
        *   **关注**: `Wallet` 的公共方法是否对其接收的参数（特别是来自 `services` 层的参数）进行了充分的校验，还是假设上层已经校验过并直接透传给底层模块（如 `Manager`, `*Store`）？
    *   **查询放大**:
        *   **关注**: `Wallet` 中所有 `get*` 或 `list*` 类型的方法，是否在内部调用了无 `limit` 的数据库查询？是否将多个小的查询聚合成一个大的、高成本的查询？
    *   **状态变更的原子性**:
        *   **关注**: `Wallet` 中涉及写操作的方法（如 `createTransaction`, `broadcastTransaction`），其内部的多个步骤（如创建、签名、入池、广播）是否是原子性的？异常路径下是否存在状态不一致的风险？
    *   **异常封装**:
        *   **关注**: `Wallet` 是否捕获了底层的详细异常（如 `BalanceInsufficientException`），并将其重新封装为更通用的、信息量更少的异常？这是否可能向上层隐藏了重要的安全或状态信息？

### 3.8 `common` & 通用工具库

*   **模块职责**: 提供被整个项目广泛复用的基础工具类，如字节操作、序列化、集合工具等。
*   **主要审计路径**:
    *   `common/src/main/java/org/tron/common/`
*   **核心审计点**:
    *   **字节与序列化工具**:
        *   **检查点**: `ByteArray`, `RLP`, `JsonUtil`
        *   **关注**: 字节数组操作（如 `copyOfRange`）是否存在边界检查问题？自定义的序列化/反序列化逻辑是否存在 DoS 风险（如整数溢出、无限递归）？
    *   **数据结构**:
        *   **检查点**: 自定义的数据结构，如 `LRUCache`。
        *   **关注**: 自实现的缓存是否有并发安全问题？过期和淘汰策略是否健壮？是否存在内存泄漏的可能？
    *   **数学与密码学工具**:
        *   **检查点**: `Hash`, `SignUtils`, `ECKey`
        *   **关注**: 这些底层工具类是否正确处理了异常输入？是否存在因错误使用密码学库而导致的安全问题（如不安全的随机数源、密钥泄露风险）？

### 3.9 `protocol` & 数据契约

*   **模块职责**: 使用 Protobuf 定义了整个系统的核心数据结构，是模块间通信的数据契约。
*   **主要审计路径**:
    *   `protocol/src/main/protos/`
*   **核心审计点**:
    *   **无界列表 (`repeated`)**:
        *   **关注**: 消息定义中，`repeated` 字段是否被广泛使用？上层模块在使用这些消息时，是否对 `repeated` 字段的 `size()` 做了严格的上限检查？（例如，一个交易包含无限个 `Contract`）
    *   **可选与必需字段 (`optional` vs `required`)**:
        *   **关注**: 核心业务逻辑是否依赖于某个 `optional` 字段的存在？如果攻击者不提供这个字段，是否会导致空指针异常或逻辑绕过？
    *   **数据类型选择**:
        *   **关注**: 是否使用了可能导致溢出或精度问题的数值类型？对于需要精确计算的金额等字段，是否使用了安全的数据类型？

### 3.10 `plugin` & 插件化架构

*   **模块职责**: 提供插件化能力，允许节点加载外部功能。
*   **主要审计路径**:
    *   `plugins/`
    *   `framework/src/main/java/org/tron/core/services/event/` (事件订阅是插件的一种形式)
*   **核心审计点**:
    *   **插件加载与权限**:
        *   **关注**: 插件是如何被加载的？加载机制是否安全？插件是否运行在一个受限的沙箱环境中？
    *   **资源隔离**:
        *   **关注**: 插件是否能无限制地访问节点的 CPU、内存、网络或数据库资源？单个插件的崩溃或恶意行为是否会导致整个节点崩溃？
    *   **数据泄露**:
        *   **关注**: 插件是否能访问到不应暴露给它的敏感数据（如私钥、节点内部状态）？
    *   **事件订阅滥用**:
        *   **检查点**: `EventPluginLoader`, `EventSubscribeService`
        *   **关注**: 事件订阅的回调是否是同步阻塞的？如果插件的回调执行缓慢，是否会阻塞主线程？事件队列是否有积压和内存溢出的风险？
