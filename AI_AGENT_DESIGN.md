# AI 代理设计文档 (AI Agent Design)

本文档旨在阐述本项目中 AI 代码审查代理的设计理念、核心组件、最佳实践与实现原理，作为团队使用和维护 AI 代理的统一指南。

---

## 第一部分：核心设计理念 (Core Design Philosophy)

为了构建一个强大、可维护且可扩展的 AI 审查系统，我们采用“关注点分离”的设计原则。AI 的能力被分解为两个层面：

1.  **全局知识与角色 (Knowledge & Persona)**: AI 在项目中的通用背景知识和扮演的专家角色。
2.  **具体任务与技能 (Task & Skill)**: AI 执行某个具体、可复用任务的标准流程，这些技能被注册为可供 AI 自主选择的 **工具**。

`GEMINI.md` 和 `Skill` 文件正是这一理念的具体实现。

---

## 第二部分：两大核心组件 (The Two Core Components)

| 特性 | `GEMINI.md` | `Skill` (`SKILL.md`) |
| :--- | :--- | :--- |
| **目的** | 定义 **全局角色** 和 **项目背景知识** | 定义一个可被 AI 调用的 **具体工具 (Tool)** |
| **作用域** | **项目级** (作用于当前目录所有 `gemini` 命令) | **工具级** (在 `gemini-cli` 中注册为一个可用工具) |
| **加载方式** | **自动加载** (只要文件存在) | **自动注册** (只要文件在 `skills` 目录下) |
| **调用方式** | (不适用) | **由模型根据 `prompt` 的意图自主选择调用** |
| **比喻** | AI 的 **“岗位说明书”** 或 **“大脑知识库”** | 工具箱中一件件 **“标准作业工具 (SOP)”** |

### 2.1 `GEMINI.md`: AI 的“大脑”——定义“我是谁”和“我在哪”

`GEMINI.md` 是项目的“全局设定”文件。它为在该项目中运行的所有 `gemini` 命令提供了一个统一的、持久的上下文和行为准则。

*   **核心职责**:
    *   **角色定义 (Persona)**: 明确 AI 的专家身份。例如：“你是一个区块链全节点安全审计专家（Java）...”。
    *   **项目背景知识 (Project Context)**: 提供项目的高层架构描述和领域知识。例如：“本项目是 `java-tron`，核心模块包括 `p2p`, `consensus`, `db`...”。
    *   **全局审核原则 (High-Level Principles)**: 设定适用于所有任务的根本原则。例如：“必须始终采用攻击者视角”、“资源耗尽模型优先”等。

### 2.2 `Skill` (`SKILL.md`): AI 的“工具箱”——定义“我能用什么”

`Skill` 文件定义了一个个具体、独立的 **工具**，供 AI 在解决问题时自主选择和使用。

*   **核心职责**:
    *   **工具描述 (Description)**: 在 `SKILL.md` 中清晰地描述该工具的用途，这是 AI 能否做出正确决策的关键。
    *   **任务目标与步骤 (Checklist)**: 定义该工具被调用时，AI 需要执行的具体任务和详细步骤。
    *   **输出格式 (Formatting)**: 严格规定 AI 输出内容的格式，便于机器解析或人工阅读。

## 第三部分：最佳实践与指南 (Best Practices & Guidelines)

本部分提供日常使用 AI 代理的“How-To”手册，指导如何正确、高效地与其协作。

### 3.1 实现稳定的审查结果

大语言模型的非确定性是其特性，但在自动化流程中，我们需要稳定的结果。请组合使用以下策略：

1.  **控制模型参数**: 在工作流的 `settings` 中，将 `temperature` 设为 `0.0`，`top_k` 设为 `1`。这是实现确定性输出最关键的一步。
    ```yaml
    # in gemini-review.yml
    # ...
        with:
          settings: |-
            {
              "model": {
                "temperature": 0.0,
                "top_k": 1,
                "top_p": 1.0
              },
              # ...
            }
    ```

2.  **强化 Skill 指令**: 在 `SKILL.md` 中使用强硬指令（如“必须”、“严格按照”）并提供一个完整的“输出范例”(Few-shot Prompting)，减少模型自由发挥的空间。
    ```markdown
    # In SKILL.md
    ## [强制] 审计输出格式
    ...
    ### 范例：
    #### ⚠️ 风险点: 并发环境下对非线程安全 Map 的访问
    *   **问题描述**: `CacheManager` 类中的 `userCache` 字段使用了 `HashMap`...
    *   **修复建议**: 建议将 `private Map<String, User> userCache = new HashMap<>();` 替换为 `private Map<String, User> userCache = new ConcurrentHashMap<>();`。
    ```

3.  **锁定模型版本**: 在工作流中指定一个具体的模型版本号（如 `gemini-1.5-pro-001`），而非 `latest`，以消除因模型自动升级带来的不确定性。
    ```yaml
    # in gemini-review.yml
    # ...
        with:
          gemini_model: 'gemini-1.5-pro-001' # 使用一个固定的模型版本
          # ...
    ```

### 3.2 应对大型 PR 的挑战

当 PR 过大时，可能会出现 Skill 激活失败或 AI 指令遵循能力下降的现象。这不是系统故障，而是受限于模型有限的 **上下文窗口 (Context Window)**。

*   **根本原因：注意力稀释 (Attention Dilution)**: 过长的代码差异会占据模型大部分“注意力”，导致分配给 Skill 定义或 `GEMINI.md` 中关键指令的权重降低，甚至在处理过程中“忘记”了它们（“迷失在中间”现象）。
*   **具体表现**:
    1.  **Token 超限截断**: CLI 工具为防止 API 报错而截断输入，可能丢失关键信息。
    2.  **工具使用决策失效**: 过多无关信息会干扰模型判断“是否需要调用某个工具”的准确性。
    3.  **会话轮次限制 (`maxSessionTurns`)**: 在多轮交互中，最初的 Skill 上下文可能会被海量的代码“挤出”模型的有效记忆区。
*   **缓解策略**:
    *   **鼓励小型、专注的 PR**: 这是最有效的解决方案，也是软件工程的最佳实践。
    *   **使用更直接的 Prompt**: 在处理大型 PR 时，使用“按名调用”或反复强调审查重点的 Prompt，有助于将模型的注意力拉回到核心任务上。

### 3.3 编写高效的 Skill

1.  **优化 Skill 描述**: 这是提高 Skill “命中率”最有效的方法。确保 `SKILL.md` 文件中的描述足够清晰、准确，并包含丰富的关键词，以便模型能更好地理解该工具的用途，从而在需要时准确地调用它。
    *   **示例**: 如果一个 Skill 的描述是“用于分析 Python 性能”，但一次 PR 提交全是 Markdown 文档的修改，模型会正确地判断不需要调用该技能，从而跳过。

2.  **改进 Prompt (提示工程)**: 如果您想在执行一个主要任务的同时，鼓励模型考虑使用其他工具，可以在 `prompt` 中进行引导。
    *   **示例**: `prompt: "请使用你所有相关的专业技能，对本次PR进行一次全面的 /gemini-review 审查。"` 这样的指令在给出直接命令的同时，也提醒模型去审视它的整个工具箱。

## 第四部分：实现原理与排查 (Implementation & Troubleshooting)

本部分提供“深入了解”和“解决问题”的参考，供维护者和高级用户查阅。

### 4.1 工作流中的实现原理

1.  **工具注册**: `gemini-cli` 启动时，自动扫描 `.gemini/skills/` 目录，并将找到的每一个 Skill **注册为一个可供模型使用的工具**。
2.  **加载知识**: `gemini-cli` 自动加载项目根目录下的 `GEMINI.md`，让 AI “角色上身”，并装载好项目知识。
3.  **模型决策**: 工作流通过 `prompt` 参数提供用户指令。AI 模型会分析这个 `prompt` 的 **意图**，并查看它手中所有可用的工具（已注册的 Skills），自主决定是否需要以及使用哪个工具来完成指令。
4.  **执行工具**:
    *   **按名调用 (Call by Name)**: 如果 `prompt` 的内容与一个已注册工具的名称完全相同（如 `/gemini-review`），AI 会跳过复杂的意图识别，直接执行该特定工具。这是一种强制执行特定工具的快捷方式。
    *   **基于意图的调用 (Call by Intent)**: 如果 `prompt` 是一个自然语言描述（如 "Review this PR for potential bugs"），AI 则会根据这个意图，在它所知的工具中选择最匹配的一个来执行。

### 4.2 Skill 激活失败排查指南

1.  **检查 Action 日志**: 确保工作流中设置了 `gemini_debug: true`。在 GitHub Action 的运行日志中，搜索 **`[Skill Discovery]`** 或 **`[Tool Call]`** 等关键字。这可以帮助您看到模型是否尝试过发现或调用该 Skill，以及是否因为匹配度不够等原因而跳过。
2.  **理解 CI 环境行为**: 在 GitHub Actions 等非交互式环境中，`gemini-cli` 会自动跳过需要人工确认的步骤（静默激活），且工作流的 `permissions` 配置已完成授权。因此，激活失败通常不是权限问题。
3.  **分析 Prompt 冲突与按名调用**: 当您使用“按名调用”时，会大大减少模型自主选择其他工具的空间。如果发现某个您期望的 Skill 未被激活，请检查您的 `prompt` 是否因为是“按名调用”，导致模型没有机会进行更广泛的意图识别去调用其他工具。
